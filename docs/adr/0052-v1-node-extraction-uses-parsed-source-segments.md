# V1 node extraction uses parsed source segments

Accepted. This amends ADR-0041: full `Parsed Source Markdown` is no longer the direct LLM input for the `Node Extraction Agent Step`. The graph-authoring pipeline first derives `Parsed Source Segments` from the parsed Markdown, runs source-grounded node extraction over those bounded segment inputs to produce `Segment Node Extraction Drafts`, then uses a dedicated LLM-backed `Node Skeleton Reconciliation Agent Step` to deduplicate and merge those drafts into `Source-Grounded Node Skeletons` before downstream rubric and edge steps. This preserves source grounding while keeping node-extraction context size controllable.

**Considered Options**

- Send the full `Parsed Source Markdown` to node extraction and rely on the model to scan, merge, and stay within context limits.
- Derive `Parsed Source Segments` first, run segment-level extraction, then use a dedicated LLM-backed reconciliation step to deduplicate and merge extracted node drafts.

**Consequences**

A valid parsed source segment may return zero `Segment Node Extraction Drafts` when it contains no diagnosable concept. That is distinct from a failed segment output; parse errors, schema validation errors, and source-grounding errors remain fail-fast for the whole graph-authoring run.

If all valid parsed source segments together produce zero drafts, the graph-authoring run fails after segment draft validation because reconciliation has no valid input. When artifacts are written, the run may retain parsed segments, per-segment traces, `segment_node_extraction_drafts.json`, and `workflow_log.json`, but it must not write `source_grounded_node_skeletons.json`, candidate nodes, or candidate edges.

Node extraction becomes a segment-driven source-reading step. The initial graph-authoring run boundary remains one selected authoritative source, matching the current single-PDF/source API shape; multi-source reconciliation is deferred until it is actually needed. The pipeline persists the derived segments as `Graph Authoring Intermediate Artifacts`, initially including `intermediate/parsed_source_segments.json`, `intermediate/segment_node_extraction_drafts.json`, `intermediate/node_skeleton_reconciliation.json`, and `intermediate/source_grounded_node_skeletons.json`, and records their counts and artifact URIs in the workflow log when artifacts are written. `parsed_source_segments.json` stores each segment's full `text` for replay/debug; later draft and reconciliation artifacts keep segment ids, source locators, grounding notes, and provenance, but they do not duplicate segment text or full source text. `Segment Node Extraction Drafts` are intentionally thin: each draft contains only `draft_id`, `segment_id`, `name`, `definition`, `source_locator`, and `grounding_note`, and the extraction step must not generate rubrics, edges, type taxonomy, difficulty labels, prerequisite claims, or other downstream judgments. `source_grounded_node_skeletons.json` is the clean rubric-step input and contains only `id`, `name`, `definition`, `source_locators`, and `source_grounding_notes`; workflow code derives `id` from the reconciled canonical `name`, the skeleton `id` becomes the final `KnowledgeNode.id`, duplicate derived ids fail validation rather than receiving automatic suffixes, and supporting draft ids, supporting segment ids, and merge or split notes remain in `node_skeleton_reconciliation.json` and are not passed to the rubric prompt. Segment derivation has a deterministic validation checkpoint before any LLM call: at least one segment must exist, segment ids must be unique, and each segment must have nonblank source, location, text, source locator, and a `heading_path` of at most three levels. Segment ids are run-local sequential ids assigned in document order, for example `seg_000001`; oversized-section splits continue the same sequence, and ids must not encode heading text, hashes, or reviewer-facing source location. Segment sizing uses character length as a rough context-size proxy, segment artifacts record `char_count`, and the initial slice does not add a tokenizer dependency or provider-specific token counting. Current code defaults use large source windows for long-context LLMs: roughly `100_000` characters as the target, `150_000` as the chunk cap where paragraph boundaries permit, and `50_000` as the preferred lower bound except for naturally small final or boundary-constrained segments. Segmentation thresholds, including maximum segment character length, target character length, minimum merge size, and paragraph overlap, stay as internal code defaults in the initial slice and are not exposed as `POST /api/authoring/graph-candidates` request parameters. If segment validation fails and `write_artifacts=true`, the failed run should still write `workflow_log.json` and any produced `parsed_source_segments.json`, but it must not write downstream draft, skeleton, candidate-node, or candidate-edge files. Segment-level extraction drafts cannot be directly concatenated into the global skeleton list; a dedicated LLM-backed reconciliation step is required before `Source-Grounded Node Skeletons` are validated for rubric authoring. Segment-level extraction is fail-fast in the initial slice: if any segment output fails parsing or validation, the whole graph-authoring run fails, traces and intermediates are retained for debugging, and the workflow must not skip that segment and continue. Segment-level extraction traces should be saved at per-segment granularity, for example under `agent_traces/node_extraction/{segment_id}/model_raw_output.txt` and `parser_output.json`, or an equivalent batch-trace representation keyed by segment id. Segment-level extraction prompts should expose only simple source context, initially `Source`, `Location`, and `Text`; internal fields such as `segment_id`, local-heading metadata, and detailed locator metadata stay in workflow artifacts rather than being model-facing prompt fields. Segment-level extraction prompts should apply a whole-book draft budget: a normal large segment usually returns 3-8 drafts, 12 drafts is exceptional, and the model should keep only concepts that could plausibly belong in a reviewed whole-book graph with no more than 100 final nodes; parser or validation code should not fail solely because a segment returned more drafts. Reconciliation reads structured `Segment Node Extraction Drafts` and their source-grounding metadata, not original segment text, so it cannot become a second full-source reading step. Because that input is node-level rather than full-book text, the initial implementation uses one global reconciliation call; hierarchical or batched reconciliation is deferred until draft volume proves that one global pass is not workable. Reconciliation therefore keeps one global step trace, for example `agent_traces/node_skeleton_reconciliation/model_raw_output.txt` and `parser_output.json`; merge and split provenance lives in `intermediate/node_skeleton_reconciliation.json` rather than in per-draft or per-segment trace directories. Reconciliation is also fail-fast: if reconciliation parsing or validation fails, the run may retain parsed segments, segment drafts, reconciliation trace/parser output, and `workflow_log.json`, but it must not write `source_grounded_node_skeletons.json`, candidate nodes, or candidate edges. Reconciliation may merge same-concept drafts across chapters and may split an obviously compound draft when the resulting concepts are already supported by draft notes and locators, but it must not invent a skeleton without support from at least one draft. Reconciliation output should preserve draft provenance, including supporting draft ids, indirect segment ids, and a short merge or split note, as intermediate metadata for validation and debugging; that provenance is not copied into final `KnowledgeNode` objects. Merged skeletons must preserve multiple source locators and enough grounding notes for review. Candidate node `Source Locator` values still point to reviewer-usable positions in the original authoritative source; segment ids remain debug and replay metadata, not node locator values. The downstream `Node Rubric Authoring Agent Step` continues to receive only reconciled `Source-Grounded Node Skeletons`, their locators and grounding notes, and the global mastery scale; it does not receive segments, drafts, reconciliation provenance, or source text. The downstream `Edge Proposal Agent Step` keeps its existing input boundary over complete candidate nodes, rubrics, source locators, and grounding notes; it does not receive segments, drafts, reconciliation provenance, or source text. `POST /api/authoring/graph-candidates` keeps a high-level response shape with skeletons, candidate nodes, candidate edges, run-log summary, and artifact paths; parsed segments, segment drafts, and reconciliation provenance are inspected through intermediate artifacts rather than inlined into the HTTP response. Small source sections may be packed with adjacent sections in document order, including across heading-parent boundaries, until the coarse context window reaches the target budget or the next section would exceed the maximum chunk budget. Packed segment locators should describe the start and end source locations so reviewer-facing source trails remain usable. Oversized sections may be split by character-count budget within the same heading boundary. The segmenter currently avoids paragraph overlap to reduce duplicate node drafts; if overlap is reintroduced later, it must not cross into another section or subsection. The segmenter is source-agnostic: it should identify a shallow structural `heading_path` of at most three levels, corresponding to chapter, section, and subsection where available, preserve optional local headings separately, exclude structural noise such as tables of contents, empty text, parser chunk comments, and obvious headers or footers, and preserve locator metadata, but it should not hard-code book-specific categories such as labs or exercises. For ISLP-style MinerU Markdown where headings are flattened to the same Markdown depth, the segment hierarchy is derived from numbered heading text rather than Markdown heading level, and table-of-contents-like heading lines are excluded from body segments. The final review output remains only `candidate_nodes.json` and `candidate_edges.json`; the purpose of segmentation artifacts is context control and authoring auditability, not deep document ontology construction.
